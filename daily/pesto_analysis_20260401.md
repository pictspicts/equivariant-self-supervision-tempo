# 本家PESTOアーキテクチャの解読と現在のテンポモデルとの差異分析

自己教師ありピッチ推定モデルである「PESTO」のソースコード（`models/pesto.py`, `losses/`, `callbacks/` 等）を調査し、私たちのテンポ推定モデルがなぜモード崩壊（定数崩壊）を引き起こしているのか、その根本原因と解決策を分析しました。

## 1. カンニング不可な「アーティファクト・フリー（無劣化）」シフト
本家PESTOにおける最大の強みは、オーディオの音程（ピッチ）を変える際に**「CQT（定Q変換）の周波数ビンを配列ごと単に上下へシフト（`torch.roll`）させるだけ」**という数学的に完璧でクリーンな処理を行っている点です。

* **PESTO（ピッチ推定）**: シフト処理によって「音が不自然に歪む」などの劣化ノイズ（アーティファクト）が一切発生しません。そのため、AIは「ノイズの手掛かりからシフト量を逆算する」というカンニングができず、純粋に和声構造（ピッチ）を解析するしかありません。
* **現在の私たちの実装（テンポ推定）**: `torchaudio`のフェーズボコーダを用いた波形のタイムストレッチを行っています。この処理は、「音がペチャっと潰れる・揺れる」といった特有のアーティファクトを必ず残します。AIはテンポのビートを聴くのを放棄し、**「フェーズボコーダ特有のノイズの量から、ストレッチ倍率 $r$ を察知して適当なベースラインのテンポに掛け算して出力する」**というズルをしていました（今回起きた77.5 BPM等への定数崩壊の原因）。

## 2. 3点比較による「不変性ロス（Invariance Loss）」の強要
PESTOの学習ループ（`training_step`）では、1つのバッチ内で巧みに3つの波形を生成し、それぞれを比較しています。

1. **`x`**: そのままの綺麗な波形
2. **`xa`**: 元の波形にノイズなどを乗せたもの（ピッチは同じ）
3. **`xt`**: ピッチシフトさせ、さらにノイズを乗せたもの

そして以下の2段構えでロスを計算しています。
* **不変性ロス（$L_{inv}$）**: ノイズなしの `x` とノイズありの `xa` の出力確率分布を**「完全に一致させろ」**と交差エントロピーで強制します。これにより、AIは表面的な音色やノイズなどの情報を強制的に無視するようになります（ノイズに対する堅牢性）。
* **等変性・シフトロス（$L_{equiv}, L_{SCE}$）**: ノイズありの `xa` とシフト済みの `xt` の間でシフト量だけズレるように強制します。

私たちのモデルは「別々にストレッチ＋ノイズ処理された2つの波形」同士をいきなり比較しているため、「ノイズを無視しろ」という強い圧力が働いておらず、ノイズをヒントにしやすい相対構造になっていました。

## 3. スケールに依存しない「割り算」による等変性ロス
独自の崩壊対策として実装した「対数（$\log$）を取って引き算する等変性ロス」は、PESTO本家の実装と数学的に全く同じアプローチでした。
* **PESTO実装**: `(z2 / z1) - ratio` のように**割り算の比率**を用いてHuberロスを計算しています。
* これにより、出力確率を限界まで小さく（極小化）して引き算のロスを人為的にゼロに近づけようとするモード崩壊を完全に防いでいます。私たちの「Log-Equivariance」の設計方針は間違っていませんでした（あとはズルをさせないクリーンな波形加工を渡すだけです）。

---

## 結論と Next Step（改善プラン）
モード崩壊（特定のBPMへの定数崩壊）を根本から解決するためには、AIから**「タイムストレッチの倍率をカンニングする手掛かり（アーティファクト）」を完全に奪い去る**必要があります。

**【対応策】**
オーディオ波形に対するフェーズボコーダでのタイムストレッチ（`torchaudio.transforms.TimeStretch`）を学習プロセスから排除します。
代わりに、**「計算後のメルスペクトログラム画像を、時間軸方向に直接リサイズ（引き伸ばす/縮める）する手法」**へ切り替えます。
これは画像処理の拡大縮小と全く同じであるため、「音程を変えずにテンポだけを変える」という目的を無劣化で達成でき、フェーズボコーダ特有のノイズが一切発生しません。これにより、AIはカンニング不可能となり、純粋にビートやリズムの構造からテンポを推定するように強制されます。
